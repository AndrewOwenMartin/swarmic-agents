## 1. Recommended changes to classes, files, directories

**New File: `swarmic_agents/agent_router.py`**
*   **Class: `AgentRouter`**
    *   `__init__(agent_map: dict[str, list[str]])`: Initializes the router with a configuration mapping agent IDs to their downstream connections or "user" output targets.
    *   `get_next_agents(agent_id: str) -> list[str]`: Returns the list of agents or "user" designated as the next step for a given agent's output.
    *   `validate_connection(source: str, target: str) -> bool`: Checks if a valid path exists between two agents in the map.

**New File: `swarmic_agents/task_orchestrator.py`**
*   **Class: `TaskOrchestrator`**
    *   `__init__(watch_dir: str, router: AgentRouter, jobber_service: JobberService)`: Initializes the orchestrator with the directory to watch, the routing logic, and the existing job processing service.
    *   `start()`: Begins the directory watching loop.
    *   `_process_new_file(file_path: str)`: Triggered when a new file is detected; determines the initial agent, creates the job, and starts the chain.
    *   `_handle_job_completion(job_info: JobInfo)`: Called when a job finishes; checks the `AgentRouter` to decide if the result goes to the user or triggers a new job for a downstream agent.

**Refactor: `swarmic_agents/jobber/jobber.py`**
*   **New Class: `JobberService`** (Extract from module-level functions)
    *   Encapsulate `run_jobber`, `init_jobs`, and `write_job_db` into this class.
    *   Expose a method `submit_job(agent_id: str, payload: ChatRequest) -> JobInfo` to allow the `TaskOrchestrator` to trigger specific jobs on demand rather than relying on the global `run_jobber` loop.
    *   *Note:* Keep `Job` and `JobInfo` dataclasses as is.

**Refactor: `swarmic_agents/chat/chat.py`**
*   No structural changes needed, but `TaskOrchestrator` will directly call `send_payload` or `simple_chat` via the `JobberService` wrapper.

**New Directory: `swarmic_agents/config/`**
*   **File: `agent_connections.yaml`** (or `.json`): Stores the map of agent connections (e.g., `agent_a: [agent_b, user]`).

## 2. Justification

**Separation of Concerns (Orchestration vs. Execution)**
The current `jobber.py` handles the *execution* of individual jobs (reading inbox, sending to LLM, writing outbox). The new requirement introduces *orchestration* logic: watching a directory, chaining agents based on a graph, and deciding on output destinations. Mixing this high-level flow control into `jobber.py` would violate the Single Responsibility Principle. By creating a `TaskOrchestrator`, we isolate the "watch and chain" logic from the "process one request" logic.

**Abstraction of Job Submission**
Currently, `run_jobber` is a global loop that processes *all* jobs. For a chained workflow, we need to trigger specific jobs based on the completion of previous ones. Extracting `JobberService` allows the `TaskOrchestrator` to submit a job for `Agent A`, wait for its completion, and then programmatically submit a job for `Agent B` if the routing map dictates it. This turns the jobber from a passive batch processor into an active service component.

**Decoupling Routing Logic**
The "map of agent connections" is a dynamic configuration concern. Placing this in a dedicated `AgentRouter` class allows for:
1.  Easy validation of the graph (e.g., detecting cycles or disconnected agents).
2.  Future extensibility (e.g., adding conditional routing based on content analysis, not just static maps).
3.  Clear separation between *who* talks to *whom* (Router) and *how* the talk happens (Jobber/Orchestrator).

**File Watcher Isolation**
Directory watching is a distinct infrastructure concern. If we embed the `watchdog` or polling logic inside `jobber.py`, we couple the job processing engine to the file system monitoring. Keeping `TaskOrchestrator` separate ensures that the core job processing logic remains unit-testable without needing a file system mock for the watcher.
